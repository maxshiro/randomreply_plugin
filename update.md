# 功能相关
## 配置项
1. 添加每日凌晨1：00将reply_msg_data.csv中权重低于固定值的消息进行清理，用户可以手动设置权重阈值，默认值为0.2。
2. 添加一个开关，允许不处理私聊消息。默认不处理私聊消息，用户可以选择开启处理私聊消息的功能。
## 新功能
1. 添加一个复读功能，独立于随机回复功能。用户可以配置是否开启复读功能，以及复读的触发概率，随机逻辑和随机回复的随机逻辑相同。
2. 添加一个获取指定api图片发送的功能，独立于随机回复和上面的复读功能，用户可以配置是否开启该功能，以及触发概率，触发概率逻辑和随机回复的触发概率逻辑相同。用户可以配置一个图片api的url，机器人会从该url获取图片并发送给用户。
3. 添加一个处理`@`消息的功能，用户可以配置是否开启该功能，当机器人被@时，机器人会随机的执行随机回复、复读、获取指定api图片发送、回复固定消息等功能，用户可以配置每个功能的触发概率。

# 新逻辑添加
原来是基于用户发送消息的随机回复，现在需要添加一个基于时间戳的随机回复功能。用户可以配置是否开启该功能。

这个功能独立于原来的随机回复，用户可以选择同时开启或者只开启其中一个功能。时间戳随机回复的触发概率和原来的随机回复的触发概率逻辑相同，用户可以单独设置时间戳随机回复的触发概率。当时间戳随机回复功能开启时：

1. 每当系统时间的秒数部分为0时（即每分钟的开始），程序会根据配置的触发概率决定是否执行随机回复。
2. 如果触发随机回复，则程序会随机选择一条消息进行回复，随机选择的逻辑和原来的随机回复逻辑相同。
3. 用户可以配置时间戳随机回复的触发概率，默认值为0.01（即1%的概率触发）。用户也可以选择关闭时间戳随机回复功能，此时程序将不会在每分钟的开始进行随机回复。
4. 用户可以指定这个功能开启的时间段，比如每天的7:00-22:00之间，只有在这个时间段内，时间戳随机回复功能才会生效。

# 逻辑梳理
程序的功能归属如下：
```
时间戳随机回复功能：
    - 主随机回复
    - 随机图片api发送功能
收到消息随机回复功能：
    - 主随机回复
    - 复读功能
    - @消息处理功能
```

# 其他优化
1. 添加群组开关，用户可以配置哪些群组启用上述功能，哪些群组不启用。
2. 添加一个群聊独立的功能，每个群聊的随机回复数据库独立存储，互不干扰。比如原来的随机回复数据库是reply_msg_data.csv，现在改为reply_msg_data_{group_id}.csv，每个群聊一个文件。用户可以选择是否开启这个功能，默认不开启，开启后每个群聊的随机回复数据库独立存储，互不干扰。开启后会自动迁移原来的reply_msg_data.csv中的数据到对应的群聊文件中，迁移完成后删除原来的reply_msg_data.csv文件。用户也可以选择不迁移，直接开启后每个群聊一个新的随机回复数据库文件，原来的reply_msg_data.csv文件不再使用。


将main.py拆分为多个文件的建议结构：

**入口层** - 程序启动和依赖注入：
- main.py：AstrBot插件注册、事件绑定、依赖装配

**应用层** - 业务流程编排：
- message_processor.py：消息接收 → 验证 → 转发给各功能模块
- scheduler.py：时间戳触发、定时任务调度
- feature_orchestrator.py：协调随机回复、复读、@处理等功能的执行

**功能层** - 具体功能实现（/features）：
- random_reply.py、repeat.py、at_message.py、api_image.py

**领域层** - 业务规则（/domain）：
- message.py、randomization.py、weight_calculator.py

**基础设施层** - 数据存储和工具（/infra）：
- /repo：CSV读写操作
- /file_lock：并发安全控制
- /media：图片处理
- /remote_storage：远端存储

核心原则：**上层调用下层，同层不互相调用**。应用层不直接写CSV，通过基础设施层；功能层不处理调度，由应用层决定调用时机。
